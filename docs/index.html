Amenthyx AI Teams

65 specialized virtual engineering teams powered by Claude Code subagent orchestration.

65 Teams v4.0 5 Execution Waves 16-Section Protocol

Each team is a complete, self-contained execution protocol with specialized roles, domain-specific quality gates, wave-based parallel execution, automated PM reporting, evidence-driven execution, atomic commits, comprehensive testing, and real-time GitHub kanban tracking.

Judge Agent

PM produces 2-3 alternative plans. Judge evaluates with a 7-criterion rubric. TL gets the winning recommendation.

Code Review Agent

Automated code review gate between engineering and QA waves covering OWASP, architecture drift, and code quality.

Retrospective Agent

Post-wave analysis with metrics comparison, bottleneck identification, and learning extraction.

Dependency Guardian

Pre-release dependency audit for CVEs, license compliance, and abandoned packages.

Agent Memory

Agents write reusable learnings during execution. Future sessions load them for continuous improvement.

Cross-Team Handoff

Formal contracts for multi-team projects with API contracts, data schemas, and UI contracts.

Quick Start

Activate any team with a single command in Claude Code:

--team <teamName> --strategy path/to/strategy.md

Step 1: Create a Strategy File

Copy the strategy template and fill it out with your project brief:

cp STRATEGY_TEMPLATE.md my-strategy.md

The strategy file includes project identity, audience, prioritized features (P0/P1/P2), technical constraints, non-functional requirements, testing requirements, and evidence requirements.

Step 2: Activate a Team

--team fullStack --strategy ./my-strategy.md

Step 3: The Team Self-Organizes

  1. The activation protocol matches the team name to a team folder
  2. The strategy file is loaded as project context
  3. The Enhanced Execution Protocol is loaded
  4. The Team Leader agent boots and reads TEAM.md
  5. Agents spawn in waves: Planning, Research, Engineering, QA, Release
  6. PM creates a GitHub Project board, milestones, and issues
  7. Every agent produces evidence (screenshots, logs, test results)
  8. Atomic commits with conventional format throughout
  9. PPTX and PDF status reports generated every 6 hours

How It Works

5-Wave Execution Model

Every team follows the same wave-based parallel execution pattern:

Wave 1: Planning

PM, Judge, and TL analyze strategy, produce plans, evaluate alternatives, and create the project board.

Wave 2: Research

Research agents investigate unknowns, spike risky areas, validate technical assumptions.

Wave 3: Engineering

Domain specialists build in parallel. Code Review agent gates output before QA.

Wave 4: QA

Test agents run comprehensive test matrix. Retrospective agent analyzes metrics.

Wave 5: Release

Dependency Guardian audits, CI/CD pipeline runs, deployment with rollback plan.

Team Structure

Each TEAM.md file contains a 16-section v4.0 protocol with approximately 11 domain-specific roles, evidence tables, testing matrices, and quality gates. All teams share the Enhanced Execution Protocol and produce standardized artifacts.

Protocols

Shared protocols in the shared/ directory define cross-cutting behavior that all teams inherit.

Enhanced Execution Protocol

Core execution framework: evidence requirements, atomic commits, testing matrix, CI/CD gates.

Activation Protocol

How --team and --strategy discovery, matching, and boot sequence works.

Judge Protocol

7-criterion rubric for evaluating competing plans. Handles close-margin A/B spike execution.

Code Review Protocol

Automated review gate: OWASP checks, architecture drift detection, code quality scoring.

Retrospective Protocol

Post-wave metrics comparison, bottleneck identification, and actionable learning extraction.

Dependency Guardian Protocol

Pre-release audit for CVEs, license compliance, and abandoned package detection.

Agent Memory Protocol

Persistent learnings across sessions. Agents write findings; future runs load them.

Cross-Team Handoff Protocol

Formal contracts (API, data schema, UI) for multi-team orchestration projects.

Multi-Team Orchestration

Sequential or parallel team execution on shared workspaces with dependency management.

Rollback Protocol

Formal rollback plans with DB migration reversal, feature flags, and blue/green strategy.

A/B Plan Execution Protocol

Time-boxed spike of runner-up plan when Judge verdict margin is close.

Risk Escalation Matrix

Measurable escalation triggers with severity levels P0-P3 and response procedures.

Security Incident Response

Incident classification, containment, eradication, and post-incident review process.

Strategy Evolution Protocol

How strategies evolve during execution based on learnings and changing requirements.

Offline Mode Protocol

Run teams without GitHub access using local markdown-based kanban boards.

UAT Protocol

User Acceptance Testing procedures, sign-off criteria, and feedback loops.

CLI Tools

Python utilities in the shared/ directory for managing teams and strategies.

amenthyx_cli.py

Main CLI tool for listing, searching, composing, and validating teams.

# List all teams
python shared/amenthyx_cli.py --list

# Search for a team by keyword
python shared/amenthyx_cli.py --search "machine learning"

# Validate a team file
python shared/amenthyx_cli.py --validate 01-full-stack

strategy_validator.py

Validates strategy files before activation to catch missing sections or misconfigurations.

python shared/strategy_validator.py path/to/strategy.md

team_test_suite.py

Validates all TEAM.md files for structural consistency against the v4.0 specification.

python shared/team_test_suite.py

team_composer.py

Build custom hybrid teams by picking agents from any existing team.

python shared/team_composer.py

Strategy File Structure

Every strategy file should include these sections:

  • Project identity, audience, and value proposition
  • Feature list prioritized as P0 (must-have), P1 (should-have), P2 (nice-to-have)
  • Technical constraints and specific technology choices
  • Non-functional requirements (performance, security, scalability)
  • Testing requirements with coverage targets
  • Timeline with milestones and success criteria
  • Evidence requirements (what proof you need from the team)
  • Risk analysis and hard/soft constraints
Validate before activating: Run python shared/strategy_validator.py your-strategy.md to catch missing sections before team activation.

Contributing

Contributions to Amenthyx AI Teams are welcome. See the full CONTRIBUTING.md on GitHub.

Adding a New Team

  1. Create a new numbered folder: NN-team-name/
  2. Create TEAM.md following the 16-section v4.0 structure
  3. Include the activation keyword in the TEAM.md header
  4. Add approximately 11 domain-specific roles with responsibilities
  5. Include evidence tables, testing matrices, and quality gates
  6. Run python shared/team_test_suite.py to validate
  7. Update the README team table
  8. Submit a pull request

Team File Structure (16 Sections)

  1. Team Identity and Activation Keyword
  2. Role Definitions (~11 agents per team)
  3. Wave Execution Plan
  4. Domain-Specific Quality Gates
  5. Evidence and Proof Protocol
  6. Testing Matrix
  7. Atomic Commit Standards
  8. CI/CD Pipeline Requirements
  9. PM Reporting Cadence
  10. GitHub Project Integration
  11. Domain-Specific Tooling
  12. Error Handling and Escalation
  13. Agent Memory Hooks
  14. Cross-Team Handoff Points
  15. Rollback Procedures
  16. Success Criteria and Exit Conditions

Project Standards

  • Every repo must have a .ai/ folder with context_base.md
  • Never commit API keys to git
  • Follow conventional commit format
  • All PRs require passing team test suite validation
